## Course Material Submissions

Confirmed 2026-07-14:

- Materials optionally declare submission actions as `submissions` entries with `name` and `link` fields. The contract is defined in `Models/CourseFrontMatter.cs` and `Models/SubmissionLink.cs`; rendering lives in `Components/Pages/Blog.razor`.
- Submission actions open the relevant external form from a compact Submit dropdown beside Download. Submit uses an up-arrow/tray glyph to distinguish it from Download. Pointer hover exposes the links; click works for touch, outside-click dismisses it, and Escape closes it for keyboard users. Rendering lives in `Components/Pages/Blog.razor`; behavior lives in `wwwroot/js/submission-menu.js`.
- Google Forms are created manually per submission. Drive organization is also intentionally manual because the expected number of forms is small and manual placement handles forms with multiple file-upload questions more predictably than custom automation. Do not introduce Forms/Drive automation unless the user revisits this decision.
- Verify the contract and markup with `StaticGenerationTests` and `BlogPageTests`; verify dropdown state and dismissal with `submission-menu.test.js`. Run the full .NET gate with `dotnet test tests/Ren.Courses.Tests/Ren.Courses.Tests.csproj`.

## Native Course Material PDFs

Confirmed 2026-07-14:

- The Markdown material (YAML frontmatter plus body) is the source of PDF content. Every file in `Content/Materials` should attempt to produce a native static PDF, and the Download action should prefer that artifact. Keep `downloadLink` as an intentional external redundancy/fallback: if that material's PDF generation fails, render its existing external link instead and continue the site build. If both generation and fallback are unavailable, omit only that material's Download action and surface a prominent CI warning/step summary. Bookings, projects, events, and FAQs use separate frontmatter types and must not enter this pipeline.
- PDF generation runs during application startup immediately before BlazorStatic generation. `PdfGeneratorService` discovers non-draft Markdown files recursively from `Content/Materials` itself because BlazorStatic's post collection is not populated at that point. It publishes a per-route `PdfGenerationManifest`; `Blog.razor` resolves generated PDF, then nonblank `downloadLink`, then no Download action. A document-level failure is isolated and never stops the site build.
- `dotnet run` is the single local and CI entry point. The build target restores exact Node dependencies before Tailwind or PDF generation, while application startup bootstraps pinned Pandoc 3.10 and Tectonic 0.16.9 archives for the current Windows/Linux x64 runtime. Downloads use checksums, partial files, retries, extraction staging, and executable validation. Tectonic receives the pinned remote bundle URL and keeps its range cache under `artifacts/tectonic-cache`; it does not download the multi-gigabyte bundle eagerly.
- Frontmatter supports an optional `pdf` map with an allowlisted template name and arbitrary template variables. Templates live under `PdfTemplates/<name>/`; the selected directory is fingerprinted in full. The default template uses bundle-owned OpenType fonts, native Pandoc highlighting macros, bounded images, and table support, so output does not depend on host fonts or a host TeX installation.
- Each document fingerprint length-frames its logical inputs: generator schema/version, complete raw Markdown, selected template directory, toolchain manifest, `package-lock.json`, Mermaid configuration, referenced local media, and shared PDF assets. Therefore any corresponding frontmatter or body change invalidates that document. Cache state lives in `artifacts/material-pdfs/state`; content-addressed PDFs live in `wwwroot/pdfs`. Missing, corrupt, mismatched, inactive, or orphaned entries are invalidated/pruned, and both PDF and state publication are atomic.
- Mermaid fences are rendered as vector PDFs through the package's actual Node CLI entry point with the repository config and Puppeteer's repository-local browser cache. A Mermaid failure falls back only that document. Process execution uses `ArgumentList`, drains redirected output, distinguishes cancellation from timeout, and kills timed-out process trees.
- Generated PDFs must be excluded from SDK static-web-asset discovery in the early-imported `Directory.Build.props`; BlazorStatic copies the physical directory after startup generation. That same file excludes nested `bin`, `obj`, `tests`, `output`, `artifacts`, and `node_modules`, preventing recursive test/build output from being copied into the main output.
- CI caches toolchain/browser inputs separately from per-document PDF state and generated artifacts. The production run writes a GitHub step summary for generation/fallback outcomes. Local diagnostics and customization are documented in `README.md`.
- Verification path: delete `artifacts/pdf-toolchain`, `artifacts/tectonic-cache`, `artifacts/material-pdfs`, and `wwwroot/pdfs`, then run the documented production `dotnet run --no-launch-profile --configuration Release`. On 2026-07-14 the cold run generated all 41 PDFs with zero fallbacks in about 106 seconds; an unchanged warm run reported 41 cache hits in 0.14 seconds. The final gates passed 161 .NET tests, 111 Jest tests, 7 Python tests, and 17 Chromium material-page tests.

## Static Generation

Confirmed 2026-07-14:

- Generated article pages use `output/articles/{slug}.html`, not `output/articles/{slug}/index.html`. This was verified by generating and locally rendering a temporary material through the production entry path.
- Run local production generation with `dotnet run --no-launch-profile` after setting `ASPNETCORE_ENVIRONMENT=Production`. The repository launch profiles force Development and otherwise start a persistent server; this was confirmed by comparing the timed-out default-profile run with a successful production run.
- The home page must pass `BlazorStaticContentService<CourseFrontMatter>.Posts` into `PostsList`, which then applies `CourseContentProvider.GetVisiblePosts(sourcePosts)`. During static generation, a provider can capture the service before its post collection is populated; using its parameterless overload produced tag chips but an empty card grid. `BlogPageTests.Home_UsesParsedPostsEvenWhenProviderCapturedAnEmptyService` reproduces this lifecycle split.
- CI and local launch profiles enable `SHOWCASE_MODE`, so non-draft material remains visible outside the configured term. Local CSS and JavaScript URLs use a 12-character SHA-256 content hash; an assembly timestamp is insufficient because static-asset-only edits need not rebuild the assembly. Service-worker shell changes still require a cache-name bump. Verify generated HTML hashes against the referenced files and run `home.spec.js` in a real browser.

## Interactive Diagrams

Confirmed 2026-07-15 (marker-based placement update):

- Materials optionally declare `diagrams`, each with `title`, optional `description`, `key` (kebab-case slug), and ordered `steps`. Every step has a `title`, optional `description`, and a complete Mermaid definition in `mermaid`. The contract lives in `Models/CourseFrontMatter.cs`, `Models/LearningDiagram.cs`, and `Models/LearningDiagramStep.cs`.
- Diagrams render ONLY where the body references them with an HTML-comment marker on its own line: `<!-- diagram: key -->`. Unreferenced diagrams never render (strict mode). Same key twice produces two widget instances. Marker determines placement in both web page and generated PDF — achieved by construction: web splits rendered HTML at the verbatim comment; PDF substitutes the same marker line in body markdown before Pandoc.
- `Services/DiagramMarkers.cs` is the single syntax owner: `MarkerLine` regex for own-line matching, `FindReferencedKeys` for PDF path, `Substitute` for marker replacement, `ResolveSegments` for web path. Key format validated by `KeyFormat` regex (`^[a-z0-9]+(?:-[a-z0-9]+)*$`).
- `Components/InteractiveDiagram.razor` renders diagrams with Previous, Next, and Play controls. `wwwroot/js/interactive-diagrams.js` lazy-loads pinned Mermaid 11.16.0 only on pages with widgets, uses strict security, pre-renders every step before enabling controls, rerenders every step for site theme changes, and preserves authored source when loading or rendering fails.
- Diagram sources must be excluded from `code-features.js` and hidden synchronously before Mermaid loads; otherwise the raw definition flickers and receives an empty `.code-wrapper`. Do not trust Mermaid's generated viewBox or intrinsic `max-width`: some client renders produced a correctly sized stage with a roughly 40px drawing. Temporarily make hidden steps measurable offscreen, rebuild each SVG viewBox from `.root.getBBox()` plus padding, and enforce fill dimensions inline with `!important`; repeat synchronously whenever a step is shown. Derive the stable widget ratio from those normalized bounds.
- The 26rem-wide stage is bounded to 6rem–18rem high, so square and horizontal diagrams remain proportionate without resizing between steps. Verify YAML and static markup with `StaticGenerationTests` and `BlogPageTests`; verify controls, eager rendering, viewport normalization, stable aspect metadata, error fallback, single-step behavior, and theme refresh with `interactive-diagrams.test.js`. `interactive-diagrams.spec.js` deliberately corrupts a step to a 40px SVG with a 2000×2000 viewport and verifies repair in Chromium and Firefox.
- Content hygiene tests (`tests/Ren.Courses.Tests/DiagramContentHygieneTests.cs`) scan all `Content/Materials/**/*.md` files and fail loudly on: missing/invalid/duplicate `key` values, body markers referencing undeclared keys, and diagrams with steps that are never referenced. Runtime stays graceful (unknown-key markers produce a log warning; unreferenced diagrams are silently skipped).
- `GeneratorSchemaVersion` 3→4 (2026-07-15) because diagram placement logic changed from prepending-all-diagrams to marker-based substitution. 4→5 (2026-07-15) for float-pinning fix + fence-aware marker scanning — both change generation behavior.
- **Float pinning (2026-07-15):** Pandoc's `--from markdown` default enables `implicit_figures`, converting `![alt](path)` images into LaTeX `\begin{figure}` floats. Mid-body images drift to page top without `\floatplacement{figure}{H}`. Template `PdfTemplates/default/template.latex` now loads `\usepackage{float}` + `\floatplacement{figure}{H}` to pin figures exactly at marker position. Risk: tall diagrams on a full page cause whitespace gaps (preferable to incorrect ordering). Template directory is fingerprinted → all docs regenerate automatically.
- **Fence-aware markers (2026-07-15):** `DiagramMarkers.GetFencedRanges` computes fenced-code-block content ranges (``` and ~~~ fences). `FindReferencedKeys` and `Substitute` skip markers inside fences — prevents diagram sections from being spliced into code blocks documenting the marker syntax. Web path (Markdig HTML-escapes `<!--`) is naturally immune. Hygiene tests inherit the fix via `FindReferencedKeys`.
- **Global DiagramIndex (2026-07-15):** `ResolveSegments` uses a single global instance counter instead of per-key counts. Two distinct diagrams on one page now get sequential ids (0, 1) instead of colliding at 0. `FindDuplicateKeys` helper surfaces duplicate frontmatter keys for caller warnings; both web path (`Blog.razor`) and PDF path (`GenerateOneAsync`) log warnings on duplicates. CI backstop: `DiagramContentHygieneTests` fails build on duplicates in real content.
- **Mermaid palette (2026-07-15):** `.mmdc.json` `fontSize` 12→14; themeVariables: darker borders (`#1f2937`), tinted fills (`primaryColor #dbeafe` with `primaryBorderColor #1e40af`), strong text contrast (`textColor #1e293b`). Semantic `classDef` colors in content (red `#ef4444` / green `#16a34a`) untouched.
